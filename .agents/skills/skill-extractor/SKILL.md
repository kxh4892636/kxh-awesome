# skill-extractor

本 skill 用于从外部数据源（git 仓库、markdown 文档等）或通过交互式对话提取生成 AI skill。生成的 skill 保存到用户指定的目录下。

> **禁止使用 axiom 技能**：执行本 skill 时不得调用 `skill("axiom")`，仅遵循本文档定义的提取流程。

## 从 Git 仓库提取 Skill 的流程

1. **克隆仓库**

   首先询问用户希望将仓库克隆到哪个路径，确认后再执行克隆：

   ```bash
   git clone --depth 1 <repo-url> <用户指定的保存路径>
   ```

2. **读取文档**

   文档通常位于 `docs/` 目录下。若找不到，主动询问用户文档位置。

3. **询问 skill 保存路径**

   向用户确认生成的 skill 保存到哪个目录，确认后再执行后续操作。

4. **使用 skill-creator 编写 skill**

   调用 `skill-creator` skill 完成 skill 的编写、测试和迭代优化。`SKILL.md` 中包含该 skill 如何重新拉取 git 仓库，进行 skill 内容更新的步骤。

5. **渐进式披露**

   遵循 skill-creator 中定义的渐进式披露原则，采用三级加载结构：
   - **元数据层**（name + description）：始终在上下文中，约 100 词以内
   - **SKILL.md 主体**：skill 触发时加载，控制在 500 行以内
   - **按需资源**（scripts/、references/、assets/）：根据实际需要加载，无行数限制

   组织规则：
   - SKILL.md 保持精简，接近 500 行时增加层级并给出明确的后续指引
   - 大型参考文件（>300 行）应包含目录索引
   - 支持多领域/框架时按变体分目录组织（如 references/ 下按 aws.md、gcp.md、azure.md 拆分）
   - 在 SKILL.md 中明确引用资源文件，并说明何时需要读取它们

   参考 skill-creator 的 "Progressive Disclosure" 章节了解完整细节。

6. **数据完整性**

   这是必须遵守的硬性约束。尽管采用渐进式披露的分层结构，Skill 必须完整包含 git 仓库中文档里的**所有**内容——不得有任何一个段落、代码示例、配置项或说明的遗漏。

   渐进式披露只是"何时展示"的策略，而非"是否包含"的取舍。大文档按主题拆分到 references/ 下的多个文件，但原始文档中的每一项内容都必须在 skill 的某处可被找到。

## 从 Markdown 文档提取 Skill 的流程

从用户提供的 Markdown 文档中提取结构化信息并生成 skill。

1. **获取 Markdown 文档**

   向用户确认 Markdown 文档的来源，支持以下三种方式：
   - **本地文件路径**：用户提供文档在本地文件系统中的路径，直接读取文件内容
   - **URL 链接**：用户提供在线文档的 URL，通过 WebFetch 工具抓取内容
   - **直接粘贴**：用户在对话中直接提供 Markdown 文本内容

   若用户提供的方式不明确，主动询问并确认。

2. **读取与解析文档**

   读取文档全文内容，识别文档的结构和主题划分：
   - 提取文档的标题层级（h1-h6），理解章节组织
   - 识别文档中的代码块、表格、列表等结构化内容
   - 标注文档中的关键概念、规则、约束和示例
   - 若文档内容不完整或存在歧义，主动向用户确认

3. **询问 skill 保存路径**

   向用户确认生成的 skill 保存到哪个目录，确认后再执行后续操作。

4. **使用 skill-creator 编写 skill**

## 与用户交互式对话提取 Skill 的流程

通过多轮交互式对话，逐步梳理需求并生成 skill。

1. **明确目标与场景**

   询问用户该 skill 的用途、目标用户群体以及典型使用场景，确保理解业务背景。

2. **收集核心知识点**

   通过对话引导用户列出该 skill 需要覆盖的核心概念、规则、约束和最佳实践。可询问：
   - 有哪些必须包含的规范或规则？
   - 常见的错误模式或反模式是什么？
   - 有没有示例代码或模板可以参考？

3. **梳理交互方式**

   确认 skill 的触发条件、输入输出格式以及是否需要多步骤推理。例如：
   - 用户通常在什么情况下会调用这个 skill？
   - 期望 skill 返回什么样的结果（代码片段、检查清单、诊断报告等）？

4. **询问 skill 保存路径**

   向用户确认生成的 skill 保存到哪个目录，确认后再执行后续操作。

5. **使用 skill-creator 编写 skill**

   将对话中梳理出的内容整理为结构化输入，调用 `skill-creator` skill 完成 skill 的编写、测试和迭代优化。

6. **渐进式披露**

   遵循 skill-creator 中定义的渐进式披露原则，采用三级加载结构：
   - **元数据层**（name + description）：始终在上下文中，约 100 词以内
   - **SKILL.md 主体**：skill 触发时加载，控制在 500 行以内
   - **按需资源**（scripts/、references/、assets/）：根据实际需要加载，无行数限制

   组织规则：
   - SKILL.md 保持精简，接近 500 行时增加层级并给出明确的后续指引
   - 大型参考文件（>300 行）应包含目录索引
   - 在 SKILL.md 中明确引用资源文件，并说明何时需要读取它们

   参考 skill-creator 的 "Progressive Disclosure" 章节了解完整细节。

7. **数据完整性**

   这是必须遵守的硬性约束。尽管采用渐进式披露的分层结构，Skill 必须完整包含对话梳理出的**所有**内容——不得有任何一个知识点、规则、示例或约束的遗漏。

   渐进式披露只是"何时展示"的策略，而非"是否包含"的取舍。对话中产出的每一项内容都必须在 skill 的某处可被找到。

8. **验证与迭代**

   编写完成后，通过实际用例验证 skill 的效果，并根据反馈持续优化。
