# Note Creator

创建或优化第一性原理驱动的技术笔记。目标是只保留本质概念与关键信息，用统一结构输出高密度 Markdown。

## 模式选择

- **创建模式**：用户要求写笔记、整理知识点、总结技术文档、创建学习笔记，或提供源材料让你归纳;
- **优化模式**：用户提供已有笔记，并要求优化、改进、精简、重构、润色、按规范整理;
- **文件模式**：用户指定文件路径时，直接读取并修改文件；保留已有 frontmatter 中的 `id` 等元数据;
- **对话模式**：用户未指定文件路径时，直接输出最终 Markdown，除非用户要求先给提纲或差异说明;

优化模式的核心约束是**知识点零丢失**：原笔记中每个独立知识点在优化后必须存在。可以删除废话、重复表述、过渡语句，但不能删除、改写或新增实质知识内容。

## 优化约束

1. **知识点零丢失**：原笔记中每个独立知识点在优化后必须存在，不可删除任何知识内容;
2. **只改形式不改内容**：优化的是表达方式、格式结构、语言风格，不是知识本身;
3. **可删冗余不可删信息**：可以删除废话、重复表述、过渡语句，但不能删除任何实质知识点;

## 核心原则

1. **第一性原理**：只记录不可再拆解的核心概念和关键行为，不记录可推导的结论;
2. **信息密度最大化**：每个 bullet 承载一个独立知识点，不写废话;
3. **关键词前置**：重要术语/概念放在 bullet 开头，描述跟在后面;
4. **中英混排**：技术术语保留英文原文，描述用中文;
5. **形式服务检索**：标题、表格、代码注释都应让后续复习能快速定位知识点;

## 优化维度

### 1. 语言精简

将完整句子压缩为电报体短语，每条 bullet 以分号 `;` 结尾。

| 不要                                                                                         | 要                                                                   |
| -------------------------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| Date.parse() 是一个静态方法，它接收一个日期字符串参数，然后返回对应的毫秒级时间戳            | Date.parse(dateString) 静态方法: 返回 dateString 对应的毫秒级时间戳; |
| 在浏览器环境中，window 对象实际上是 global object 的一个代理对象                             | 浏览器中 global object 的代理;                                       |
| 下面我们来看一下如何获取 Global object                                                       | ##### 获取 Global object                                             |
| 如果鸡蛋没碎，i=i，j=j-1，最多可以检查 dp[i][j-1] 层楼层                                     | 如果鸡蛋没碎, i=i, j=j-1, 最多检查 dp[i][j-1] 层;                    |
| 这个问题的思路类似于正则表达式匹配                                                           | 思路类似于正则表达式匹配;                                            |
| 由于数组存在负数，正数与负数相乘，会让最大乘积变最小乘积，反之依然，故维护数组最大值和最小值 | 数组存在负数, 正负相乘会翻转大小, 故维护 max 和 min 两个 dp;         |

### 2. 格式规范化

- bullet 结尾统一为分号 `;`（非句号、非无标点）;
- 关键词/术语前置，描述紧跟其后;
- API 签名统一为 TypeScript 风格：`Type.method(param: Type): ReturnType: 描述;`;
- 代码块添加语言标记;
- 图片使用相对路径：`![alt](images/filename.png)`;
- 数学公式使用 LaTeX：行间 `$$...$$`，行内 `$...$`;

### 3. 结构优化

- 标题层级规范：H1(标题) → H2(大主题) → H3(子主题) → H4(子主题/具体概念) → H5(具体概念/API 分组)，最多到 H5;
- 层级必须连续递进，禁止跳级（如 H3 直接跳 H5 应改为 H3 → H4）;
- 同一标题下若存在“概念 / 定义 / 本质”类子标题，通常放在第一个子标题位置，再放语法、命令、示例、机制、边界;
- 删除引言/总结/过渡段落;
- 删除“下面介绍...”等引导语;
- 删除重复标题已表达的信息;
- 合并同一知识点的多种表述为一种;
- 将散落的同类信息归入同一 H5 分组（如：将散落的方法签名归入“实例方法”分组）;
- 避免低检索价值标题，如大量重复的“示例”“代码示例”，应改为更具体的“语法格式”“命令”“目录结构”“分类”“对比”“机制”等;

### 4. 表格化

以下场景将 bullet 转为表格：

- 属性/方法枚举（概念性）;
- 对比（如 TCP vs UDP）;
- 协议/规范要求;
- 运算符/表达式枚举（独立于代码示例时）;

禁止表格化的场景：

- 与代码强相关的并列知识点，如关键字含义、修饰符语义、类型系统枚举、工具类型清单;
- 若每条知识点对应一行可演示的代码，首选“代码块 + 注释”形式;
- 此类知识点应转入代码注释化维度;

### 5. 代码注释化

将与代码强相关的并列 bullet 知识点合并入代码块的注释，按用途选择注释位置：

- 知识讲解：概念定义、语法说明、关键字含义，使用单独一行注释，放在被解释代码上方;
- 代码解释：行为说明、运行结果、错误提示，使用行内注释，放在代码末尾;

优化前：

```markdown
- `boolean`: 布尔值
- `number`: 数值
- `string`: 字符串

\`\`\`typescript
const isDone: boolean = false;
const num: number = 1;
const myName: string = "Tom";
\`\`\`
```

优化后：

````markdown
```typescript
// 布尔值
const isDone: boolean = false;
// 数值
const num: number = 1;
// 字符串
const myName: string = "Tom";
```
````

优化前：

```markdown
- handleEvent 报错, event 不能为 'dblclick';

\`\`\`typescript
handleEvent(document.getElementById("world"), "dblclick");
\`\`\`
```

优化后：

````markdown
```typescript
handleEvent(document.getElementById("world"), "dblclick"); // 报错, event 不能为 'dblclick'
```
````

### 6. 代码块精简

- 删除非必要的代码行;
- 删除已由注释承载的冗余说明;

### 7. 结构化代码块注释

- 目录树、文件结构、配置片段等结构化代码块，若需要解释每一项作用，优先使用行内注释承载说明;
- 同一代码块内的行内注释应与主体内容保留空格，并尽量让注释列左对齐，便于扫描;
- 目录树行内注释已经解释目录/文件作用时，不再额外写重复的职责表或 bullet;

```text
project/
├── cmd/         # 可执行程序入口集合
├── internal/    # 仅当前项目内部可导入的代码
└── go.mod       # module 配置文件
```

## 文件结构

```markdown
---
id: <uuid>
---

# 标题

## 大章节 (H2)

### 子章节 (H3)

#### 子章节 (H4)

##### 具体概念 (H5)

- 知识点;
  - 补充细节;
```

### 标题层级规则

| 层级 | 用途              | 示例                   |
| ---- | ----------------- | ---------------------- |
| H1   | 文档标题，仅一个  | `# 基本引用类型`       |
| H2   | 大主题分区        | `## The Global Object` |
| H3   | 子主题            | `### Window Object`    |
| H4   | 子主题/具体概念   | `#### 实例属性`        |
| H5   | 具体概念/API 分组 | `##### 静态方法`       |

层级必须连续递进，禁止跳级（如 H3 直接跳 H5）。H4 和 H5 均可作为最细粒度标题，根据内容深度选择。最多使用到 H5。

## 内容格式规则

### Bullet Points

- 每条以分号 `;` 结尾（非句号、非无标点）;
- 不写完整句子，用电报体短语;
- 缩进 sub-bullet 补充细节或例外情况;
- 一条 bullet = 一个知识点，不合并不同知识点;
- 关键词/术语放在 bullet 开头，描述紧跟其后;

```markdown
- 无法显式访问;
- 所有可以被全局访问的变量和函数为其属性;
```

### 表格

用于对比、枚举、属性列表等结构化数据：

```markdown
| 属性    | 描述 |
| ------- | ---- |
| Math.PI | PI   |
| Math.E  | e    |
```

常见表格场景：

- 属性/方法枚举（概念性）;
- 对比，如 TCP vs UDP、01背包 vs 完全背包;
- 协议/规范要求;
- 运算符/表达式枚举（独立于代码示例时）;

禁止表格化的场景：

- 与代码紧密相关的并列知识点，如关键字含义、修饰符语义、类型系统枚举;
- 代码示例内部的语法点说明，应作为代码注释而非表格;
- 若每条知识点对应一行可演示的代码，首选“代码块 + 注释”形式;

### 代码块

- 添加语言标记，如 `typescript`、`go`、`python`、`java`、`sql`、`css`、`bash`;
- 只展示最小必要示例;
- 注释承载与代码强相关的知识点;

#### 代码注释规则

| 注释类型     | 用途                           | 位置                  |
| ------------ | ------------------------------ | --------------------- |
| **代码解释** | 行为说明、运行结果、错误提示   | 行内（代码末尾 `//`） |
| **知识讲解** | 概念定义、语法说明、关键字含义 | 单独一行（在代码上方） |

代码解释示例：

```typescript
let pattern1 = /at/g; // g 匹配全部内容
handleEvent(document.getElementById("world"), "dblclick"); // 报错, event 不能为 'dblclick'
let carProps: keyof Car; // 'manufacturer' | 'model' | 'year'
```

知识讲解示例：

```typescript
// 布尔值
const isDone: boolean = false;
// 私有, 仅能从声明类内部访问
private name;
```

多个并列且与代码强相关的知识点，必须以“代码块 + 单独一行注释”合并展示，禁止在代码外再用 bullet 重复列出：

```typescript
// 所有属性变为可选; { [P in keyof T]?: T[P]; }
type PersonPartial = Partial<Person>;
// 所有属性变为必选; { [P in keyof T]-?: T[P]; }
type RequiredContact = Required<Contact>;
// 所有属性变为只读; { readonly [P in keyof T]: T[P]; }
type ReadonlyPerson = Readonly<Person>;
```

### API 签名

使用 TypeScript 风格的类型标注：

```markdown
- ClassName.method(param: Type): ReturnType: 描述;
- ClassName.property: Type 属性: 描述;
```

示例：

```markdown
- Number.isInteger(number: number): boolean 静态方法: 判断 number 是否为 int 类型;
- String.length: number 属性: 返回字符串 code unit 数量;
```

### 图片

使用相对路径引用图片，放在对应标题下方：

```markdown
![alt](images/filename.png)
```

### 数学公式

行间公式用 LaTeX `$$...$$`，行内公式用 `$...$`：

```markdown
$$
dp[i] = \begin{cases}
nums[i] &if\ dp[i]<0 \\
dp[i-1]+nums[i] &if\ dp[i]\geq 0
\end{cases}
$$
```

## 创建流程

1. 读取所有源材料，提取标题、核心概念、关键 API、机制流程、例外情况、代码示例和表格行，建立覆盖清单;
2. 将覆盖清单映射到目标章节，避免只保留主线而遗漏边界规则、术语定义、对比项和注意事项;
3. 按逻辑关系组织标题层级（H2 → H3 → H4 → H5）;
4. 每个概念提炼为最短 bullet，必要时附带类型签名;
5. 对比/枚举类信息用表格（除非与代码强相关）;
6. 与代码强相关的并列知识点合并到代码块的单独行注释中;
7. 代码运行结果、错误、行为说明用行内注释;
8. 需要展示用法时使用最小代码块;
9. 检查并删除所有可以不写的字词，直到无法再删;
10. 对照覆盖清单做完整性校验，确认源材料中的核心知识点都有对应落点;

## 优化流程

1. **提取知识点**：逐条提取原笔记中所有独立知识点，建立清单;
2. **优化语言**：将完整句子压缩为电报体短语，统一分号结尾;
3. **规范格式**：统一 API 签名、代码块标记、图片路径、数学公式;
4. **优化结构**：调整标题层级，删除过渡段落，合并同类信息;
5. **表格化**：将适合的概念性 bullet 转为表格，排除与代码强相关的内容;
6. **代码注释化**：将与代码强相关的并列 bullet 合并入代码块注释;
   - 知识讲解 → 单独一行注释（在被解释代码上方）;
   - 代码解释 → 行内注释;
7. **精简代码**：删除非必要的代码行，不重写代码逻辑;
8. **完整性校验**：逐条对照步骤 1 的知识点清单，确认每个知识点在优化后均存在;

### 完整性校验方法

优化完成后执行以下校验：

1. 读取原笔记或全部参考材料，提取所有独立知识点（标题、bullet、代码示例、表格行、图片引用、重要链接均可作为知识点）;
2. 逐条检查新笔记或优化后笔记是否包含该知识点，或是否被合并到等价概念、表格或代码注释中;
3. 如有遗漏，将遗漏知识点补充到最合适的章节，不为补充而新建孤立结构;
4. 重复检查直到零遗漏;

## 禁止事项

- 不写引言/总结/过渡段落;
- 不写“下面介绍...”等引导语;
- 不重复标题已表达的信息;
- 不写显而易见的内容;
- 不加 emoji;
- 不写完整句子（用短语片段）;
- 不对同一知识点做多种表述;
- 不在代码块中写冗余注释;
- 不删除已有笔记中的任何知识点（优化模式）;
- 不改变已有知识点的技术含义（优化模式）;
- 不添加原笔记中不存在的新知识点（优化模式）;
- 不重写原笔记中的代码逻辑（优化模式）;
- 不将与代码强相关的并列知识点用表格或代码外 bullet 列出;
- 不混用注释位置（代码解释统一用行内，知识讲解统一用上方独立行）;
- 不跳级使用标题（H3 不能直接跳 H5）;
