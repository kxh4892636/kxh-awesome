# fe-tech-doc

## 快速开始

### 两种信息收集模式

本 skill 支持两种互补的信息收集模式，可根据用户情况灵活选择或组合：

- **模式 A（文档驱动）**：用户直接提供产品文档、后端技术文档等完整信息
- **模式 B（交互式头脑风暴）**：用户信息不完整时，通过多轮对话与用户头脑风暴，结合代码上下文逐步明确需求

判定规则：
- 用户一次性提供了产品文档/后端文档链接 → 优先走模式 A
- 用户仅有零散信息（需求描述、代码位置、Figma 等）或明确要求"头脑风暴/一起讨论/帮我梳理方案" → 走模式 B
- 两种模式可以混合：即使有产品文档，关键技术选型不明时也可以局部进入头脑风暴

### 可选信息

用户可能提供以下信息：
1. **产品文档链接**：飞书云文档链接
2. **后端技术文档链接**：飞书云文档链接  
3. **Figma 设计稿链接**：设计稿链接
4. **静态资源 URL**：图片、图标等资源的 URL 列表
5. **代码路径**：支持多种形式：
   - 前端项目根目录路径
   - 具体组件目录路径
   - 具体函数/工具文件路径
   （默认使用当前工作目录）
6. **飞书输出位置**：目标飞书文档链接（更新已有文档）或飞书知识库/文件夹位置（创建新文档）
7. **技术文档名称**：飞书文档标题，使用 `【yyyymm】-【业务需求名称】-【前端技术文档】`
8. **需求描述（模式 B）**：自然语言描述的功能诉求、业务背景、目标用户等

### 执行步骤

```
1. 判定模式 A / B / 混合
2. 信息收集：
   - 模式 A：询问缺失信息 → 使用 lark-cli 获取飞书文档
   - 模式 B：使用 AskUserQuestion 多轮交互式头脑风暴
3. 分析仓库代码和用户提供的代码路径
4. 使用 lark-doc-quality 的业务需求前端技术文档模板生成飞书富文本结构内容（必要时回到步骤 2 补充信息）
5. 使用 lark-cli 创建或更新飞书文档
```

## 详细指南

### 第一步：信息收集

根据判定结果选择模式。

#### 模式 A：文档驱动

一次性询问用户以下信息（均为可选），然后使用 lark-cli 批量获取飞书文档内容：
- 产品文档的飞书链接：
- 后端技术文档的飞书链接：
- Figma 设计稿链接：
- 静态资源 URL 列表：
- 代码路径（项目根目录/组件目录/函数文件路径...）：
- 飞书输出位置（已有文档链接或目标知识库/文件夹）：
- 技术文档名称（格式：`【yyyymm】-【业务需求名称】-【前端技术文档】`）：

#### 模式 B：交互式头脑风暴

通过多轮 `AskUserQuestion` 与用户对话，逐步明确需求。**每轮只聚焦一个主题**，避免一次抛出过多问题。

推荐的头脑风暴轮次（按需裁剪，跳过已明确的轮次）：

1. **需求与目标**（必问）
   - 功能要解决什么问题？目标用户是谁？
   - 核心使用场景和验收标准？
   - 是否有已有产品文档/PRD 片段可参考？

2. **代码上下文锚点**（必问）
   - 在哪个项目/目录下实现？（收集代码路径）
   - 是新增模块还是改造现有模块？
   - 读取并分析相关代码，向用户确认复用点和改造点

3. **UI / 设计稿**
   - 有无 Figma 链接？若有，使用 `figma-mcp` 辅助分析
   - 若无设计稿，确认 UI 形态（PC/移动端、页面/弹窗/组件）

4. **数据与接口**
   - 后端接口是否已就绪？有无 IDL/Swagger/后端文档？
   - 未就绪时，与用户讨论预期的数据结构、Mock 策略

5. **交互与状态**
   - 关键交互流程、异常态、加载态
   - 状态管理方案（组件内 state / 全局 store / URL / 服务端）

6. **埋点与监控**
   - 关键埋点事件、上报时机
   - 错误监控、性能指标

7. **输出确认**
   - 飞书输出位置：创建新文档还是更新已有文档
   - 文档标题：`【yyyymm】-【业务需求名称】-【前端技术文档】`
   - 是否需要先在当前对话中预览大纲后再写入飞书

**头脑风暴原则**：
- 每轮提问前，先基于已有信息（代码、先前回答）做出推测，给出 2-4 个候选选项，让用户在选项上做选择而非开放式填空
- 用户回答后实时总结，形成"已达成共识"的要点列表
- 关键决策保留在文档的"技术选型"章节，便于追溯
- 信息已足够生成文档时主动结束头脑风暴，不强行凑满所有轮次

### 第二步：获取飞书文档

**仅当**用户提供了飞书链接时执行。使用 `lark-cli` 获取：
- 产品文档内容（如有链接）
- 后端技术文档内容（如有链接）

模式 B 若全程无飞书文档，跳过此步骤。

### 第三步：分析代码

根据提供的代码路径类型进行分析：

**如果是项目根目录**：
1. 使用 `Glob` 和 `Read` 工具探索整体目录结构
2. 识别项目技术栈（React/Vue、TypeScript、状态管理等）
3. 了解项目代码规范和架构模式

**如果是具体组件/函数路径**：
1. 使用 `Read` 工具读取该路径下的代码文件
2. 分析组件/函数的接口设计、实现逻辑
3. 查找相关的依赖和调用关系

**如果未提供路径**：使用当前工作目录作为默认路径。

### 第四步：生成技术文档

必须使用 `lark-doc-quality` skill，并读取：

- `lark-doc-quality/SKILL.md`
- `lark-doc-quality/references/general-quality.md`
- `lark-doc-quality/references/frontend-tech-doc-template.md`

输出内容必须是适合飞书文档创建或更新的富文本结构内容，而不是 Markdown 文件。

**核心章节**：
- **工程信息**：版本记录、基本信息、POC、联调信息
- **需求分析**：需求背景、目标收益、需求列表
- **外部依赖**：TCC、网关、橙蕉、FCS/LanderX 等配置
- **数据源**：数据接口及 IDL 定义
- **功能模块**：按平台类型（PC/App/H5）划分
  - UI：设计稿链接、组件布局
  - 代码设计：组织结构、属性/事件/状态管理设计
  - 交互逻辑：用户操作、界面反馈
  - 数据逻辑：数据来源、处理、存储
  - 埋点逻辑：事件上报
  - 外部静态资源：列出图片、图标、字体、Lottie、远程资源等
- **监控方案**：新增页面、技术指标监控、业务指标监控
- **发布方案**：功能检查、数据检查、发布检查
- **Q & A**：评审问题、技术取舍、未决事项

**生成原则**：
1. 功能模块相互独立、正交，可并行开发
2. 数据接口与后端文档保持一致；模式 B 下若接口未定，按与用户讨论的结构产出并标注"待后端确认"
3. 代码变更位置必须是实际存在的路径（模式 B 下需先通过 Grep/Read 验证）
4. 无法获取的信息保留占位符并告知用户补充
5. 模式 B 下将头脑风暴中的关键决策整理到"技术选型 / 讨论结论"章节，便于追溯
6. 保留飞书富文本结构：自动编号标题、表格、checkbox、代码块、链接、引用文档、图片/设计稿/看板
7. 文档标题使用 `【yyyymm】-【业务需求名称】-【前端技术文档】`
8. 功能模块的 `技术方案` 小节统一使用表格承载，默认列为 `UI / 代码设计 / 交互逻辑 / 数据逻辑`

### 第五步：输出文档

使用 `lark-cli` 输出飞书文档：

1. 若用户提供已有飞书文档链接：先读取最新大纲和 revision，再通过 lark-cli 局部替换或追加内容，避免覆盖用户已有修改。
2. 若用户要求创建新文档：通过 lark-cli 创建飞书文档，标题为 `【yyyymm】-【业务需求名称】-【前端技术文档】`。
3. 写入后重新读取大纲或目标章节，确认标题、表格、checkbox、代码块、链接和引用文档解析正常。
4. 最终回复给出飞书文档链接、更新/创建结果、revision id；不要再要求用户去本地路径查找 Markdown 文件。
