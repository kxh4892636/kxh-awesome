# Lark Doc Quality

用于把飞书文档写成严谨、清晰、结构化、图文并茂且易于行动的文档。执行时通常与 `lark-doc` skill 配合：`lark-doc` 负责读取和写入能力，本 skill 负责文档结构、表达质量和富文本保真。

## 渐进式披露

先只读本文件，按任务类型选择需要的 reference；不要一次性加载全部模板。

| 场景 | 读取 |
| --- | --- |
| 任意飞书文档撰写、润色、改写、总结、补充 | `references/general-quality.md` |
| SOP、接入手册、操作手册、平台使用手册、流程说明 | `references/sop-template.md` |
| 业务需求前端技术文档、基于 PRD/Meego/UI/后端文档写前端技术方案 | `references/frontend-tech-doc-template.md` |

多个场景同时命中时，先读 `general-quality.md`，再读具体模板。例如“把 PRD 改成前端技术文档”需要读 `general-quality.md` 和 `frontend-tech-doc-template.md`。

## 工作流

1. 明确任务类型：读取、总结、润色、改写、补充、创建、替换、移动还是删除。
2. 明确读者和用途：面向管理者、研发、产品、运营、业务方、值班同学还是新接手成员。
3. 根据上表读取对应 reference，选择文档结构和富文本表达。
4. 使用 `lark-doc` 读取目标文档最新内容：优先 `outline`，再按章节读取相关 `section`；编辑前用 `detail full` 获取样式和 block id。
5. 读取用户提供的引用文档，提取事实、流程、截图、表格、画板、代码块、看板和平台链接。
6. 局部更新文档：优先使用现有标题、表格、提示块、画板或段落的 block id 做 `block_insert_after` / `block_replace`，避免全量覆盖。
7. 写入后重新读取大纲和目标章节，确认位置、标题层级、链接、引用、图片、表格、代码块没有丢失。

| 文档类型 | 主要目标 | 推荐结构 |
| --- | --- | --- |
| 产品/项目方案 | 对齐目标、范围、路径和风险 | 背景 / 目标 / 范围 / 方案 / 里程碑 / 风险 / 决策 |
| 技术方案 | 说明设计取舍、接口、架构和验证 | 背景 / 目标 / 方案设计 / 数据流 / 接口 / 灰度 / 测试 / 风险 |
| 业务需求前端技术文档 | 把产品需求翻译成前端可开发、可联调、可监控、可发布的工程方案 | 工程信息 / 需求分析 / 外部依赖 / 技术方案 / 监控方案 / 发布方案 / Q &amp; A |
| SOP/操作手册 | 让读者按步骤完成任务 | 总览与速查 / 流程模块 1 / 流程模块 2 / ...；每个流程模块固定包含介绍、使用流程、常见问题 |

| 调研/竞品分析 | 形成事实、洞察和建议 | 结论先行 / 方法 / 发现 / 对比表 / 洞察 / 建议 |

| 知识整理 | 把零散信息变成可复用知识 | 总览 / 核心概念 / 流程 / 示例 / 注意事项 |

## 不可省略的通用规则

- 读者优先：文档结构、语言和素材选择都要降低理解成本，帮助读者快速理解和行动。
- 标题下先定调：重要章节标题下可用高亮块或首段概述说明结论、背景、范围和阅读路径；这部分可以适当展开。
- 正文保持精简：概述之后尽量短句化、分点化，在语义不失真的前提下使用最少文字。
- 分点表达：列表项优先使用“总结 + 简短概述”，先给关键词或判断，再用一句话补充说明。
- 结论前置：章节、表格或关键段落先说明结论、定位或作用，再补充来源、细节和验证方式。
- 结构承载逻辑：对比、清单、步骤用表格；流程、链路、关系用画板；说明文字只补关键判断。
- 富文本优先：结论用 callout，关系用 whiteboard，对比用 table，步骤用 list 或 checkbox，截图贴近对应说明，来源用 cite 或链接，代码用 code block。
- 图片使用原图：复用或迁移文档图片时，始终从源文档下载或复制原始图片资源，不使用对截图再次截图、画板导出图、预览缩略图、压缩 JPG 或二次裁剪图替代；写入目标文档或画板前确认图片 token / 本地文件来自原始素材。
- 概念前置：后文依赖专业名词、业务口径、系统模块或数据指标时，先给出简短解释。
- 表格自解释：表格不只罗列字段、链接或截图，应补充定位、用途、影响或承接关系。
- 图文一致：正文、表格、画板、截图说明使用同一套分层、命名和阅读顺序。
- 引用要转化：引用文档、看板、图片或代码资料要提炼为当前文档可直接使用的结论、规则、入口或素材，并说明用途。
- 事实可追溯：关键判断要能追溯到引用文档、截图、看板、代码或明确来源；不编造负责人、时间点、权限、平台能力、数据口径、结论或引用来源。
- 样式服务内容：调整排版、表格、画板、颜色和层级时，只提升可读性，不改变事实、范围和业务含义。
- 飞书章节标题使用自动编号：标题 XML 中保留或新增 `seq="auto"` 和 `seq-level="auto"`，不要手写 `1.3`、`4.` 这类章节序号。
- 引用文档优先用飞书 cite 组件；无法 cite 时用普通链接。
- 需要内部跳转时，先读取 `outline --detail with-ids`，链接使用目标文档 URL + `#block_id`。
- 默认做局部更新，保留已有图片、画板、表格、引用、链接和章节关系；不使用 `overwrite`，除非用户明确要求重建全文。
- 写入 XML 时只转义文本里的 `<`、`>`、`&`，不要转义标签本身。

## 最终回复

完成后只汇报关键结果：

- 更新了哪个文档/章节；
- 新增或替换了什么结构；
- 是否校验通过；
- 当前 revision id；
- 如果有未验证事项，直接说明。

不要把大段文档内容复制到最终回复里，用户可以直接在飞书文档中查看。
