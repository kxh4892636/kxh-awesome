# fe-tech-doc

## 使用范围

仅用于把用户已提供的需求、方案、接口、设计或代码上下文整理为“业务需求前端技术文档”，并写入飞书文档。

除非用户明确另行要求，不执行以下步骤：
- 信息收集模式判定
- 多轮需求访谈或头脑风暴
- 主动读取产品/后端飞书文档
- 主动分析 Figma
- 主动扫描或分析代码仓库

若缺少写入飞书必需的信息，仅询问最小必要项：飞书输出位置和文档标题。

## 执行步骤

```
1. 生成飞书文档内容
2. 输出飞书文档
```

## 第一步：生成飞书文档内容

先读取 `references/template.md`。该模板是飞书文档的固定格式，同时包含各章节应该写什么内容的说明。

使用 `lark-doc-quality` skill 生成适合飞书创建或更新的富文本结构内容，而不是 Markdown 文件。调用时明确文档类型是“业务需求前端技术文档”，并要求输出结构与 `references/template.md` 一致。

生成要求：
1. 严格沿用 `references/template.md` 的标题层级、章节顺序和占位符语义。
2. 按模板中每个章节的说明填充内容；模板里的说明不是最终正文时，应转化为具体方案内容。
3. 基于用户当前已提供的信息生成，不额外扩展需求调研流程。
4. 功能模块尽量独立、正交，便于并行开发。
5. 数据接口、UI、代码设计、交互逻辑、数据逻辑、埋点逻辑、外部静态资源等内容只在已有信息支持时展开；缺失处保留 `待补充` 或 `待确认`。
6. 涉及代码路径时，只使用用户已提供或当前上下文已确认的路径；不为了补全路径而主动扫描仓库。
7. 内容应直接适配飞书富文本写入，包含合适的标题层级、表格、checkbox、代码块、链接和引用文档结构。

## 第二步：输出飞书文档

使用 `lark-cli` 创建或更新飞书文档。

1. 若用户提供已有飞书文档链接：先读取最新大纲和 revision，再局部替换或追加内容，避免覆盖用户已有修改。
2. 若用户要求创建新文档：使用用户提供的飞书知识库/文件夹位置创建文档；标题优先使用用户给定标题，未给定时使用 `【yyyymm】-【业务需求名称】-【前端技术文档】`。
3. 写入后重新读取大纲或目标章节，确认标题、表格、checkbox、代码块、链接和引用文档解析正常。
4. 最终回复给出飞书文档链接、创建/更新结果和 revision id；不要要求用户去本地路径查找 Markdown 文件。

## 依赖技能

- `lark-doc-quality`：生成飞书富文本结构内容，并明确任务类型为“业务需求前端技术文档”。
- `lark-doc` / `lark-cli`：创建、读取、更新飞书文档。

## 参考资料

- `references/template.md`：飞书文档固定格式和各章节写作说明。生成文档前必须读取并遵循。
